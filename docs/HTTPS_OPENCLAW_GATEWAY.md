# OpenClaw Gateway — HTTPS Setup Spec

## 1. Purpose

The OpenClaw gateway dashboard UI requires a **secure context** (HTTPS or `localhost`) to use browser device-identity APIs. The gateway currently runs only on `http://10.0.0.60:18789/`. This spec covers adding an nginx TLS reverse proxy in front of it using a locally-trusted certificate generated by `mkcert`, so that `https://10.0.0.60/` works in any browser on the local network without certificate warnings.

**Implementation target:** All VM provisioning steps below must be implemented as Terraform `null_resource` provisioners in `terraform/openclaw_vm.tf`, extending the existing `null_resource.openclaw_vm_config` pattern.

## 3. Target State

- nginx installed and enabled as a systemd service
- mkcert installed at `/usr/local/bin/mkcert`
- Local CA root cert at `/etc/ssl/openclaw/rootCA.pem`
- TLS cert + key for `10.0.0.60` and `localhost` at `/etc/ssl/openclaw/`
- nginx listens on port 443, reverse-proxies to `localhost:18789`
- Port 18789 remains open (no firewall change needed; nginx is additive)
- Root CA cert installable on the user's Mac so browsers trust the cert without warnings
- Terraform outputs `openclaw_https_url` and `rootca_mac_trust_commands` for post-apply guidance

## 5. Terraform Implementation

### 5.1 New resource: `null_resource.openclaw_vm_nginx`

Add to `terraform/openclaw_vm.tf` after the existing `null_resource.openclaw_vm_config`.

**Requirements:**
- `depends_on = [null_resource.openclaw_vm_config]` — nginx setup runs only after OpenClaw is confirmed healthy
- Same `connection` block as `openclaw_vm_config` (SSH, `var.openclaw_vm_host`, `var.openclaw_vm_user`, `var.openclaw_vm_password`)
- `triggers` keyed on `vm_host` and `script_hash = filemd5("${path.module}/openclaw_vm.tf")` (consistent with existing resource)
- Single `remote-exec` provisioner using the same `bash <<'REMOTESCRIPT' ... REMOTESCRIPT` pattern already used in `openclaw_vm_config`
- The `run_sudo()` helper (`echo "$VM_PASSWORD" | sudo -S -E "$@"`) must be redefined inside this script block (it does not carry over from the previous resource)

**Script steps (in order):**

#### Step 1 — Install packages
```bash
run_sudo apt-get update -qq
run_sudo apt-get install -y nginx libnss3-tools
```

#### Step 2 — Install mkcert (idempotent guard)
```bash
if [ ! -f /usr/local/bin/mkcert ]; then
  curl -fsSL "https://dl.filippo.io/mkcert/latest?for=linux/amd64" -o /tmp/mkcert
  chmod +x /tmp/mkcert
  run_sudo mv /tmp/mkcert /usr/local/bin/mkcert
fi
```

#### Step 3 — Generate local CA and certificate
```bash
run_sudo mkdir -p /etc/ssl/openclaw
run_sudo env CAROOT=/etc/ssl/openclaw mkcert -install
run_sudo env CAROOT=/etc/ssl/openclaw mkcert \
  -cert-file /etc/ssl/openclaw/cert.pem \
  -key-file  /etc/ssl/openclaw/key.pem \
  10.0.0.60 localhost
run_sudo chmod 644 /etc/ssl/openclaw/cert.pem
run_sudo chmod 600 /etc/ssl/openclaw/key.pem
```

#### Step 4 — Write nginx config (temp file pattern)

Because `run_sudo` pipes the password via stdin, `run_sudo tee` cannot also receive a heredoc on stdin. Use a temp file instead:

```bash
cat <<'NGINXCONF' > /tmp/openclaw_nginx.conf
server {
    listen 443 ssl;
    server_name 10.0.0.60 localhost;

    ssl_certificate     /etc/ssl/openclaw/cert.pem;
    ssl_certificate_key /etc/ssl/openclaw/key.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    location / {
        proxy_pass         http://localhost:18789;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection "upgrade";
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto https;
        proxy_read_timeout 86400;
    }
}

server {
    listen 80;
    server_name 10.0.0.60 localhost;
    return 301 https://$host$request_uri;
}
NGINXCONF

run_sudo cp /tmp/openclaw_nginx.conf /etc/nginx/sites-available/openclaw
run_sudo chmod 644 /etc/nginx/sites-available/openclaw
rm /tmp/openclaw_nginx.conf
```

> **Heredoc quoting note:** The nginx config heredoc delimiter must be quoted (`<<'NGINXCONF'`) so that bash does not expand `$http_upgrade`, `$host`, etc. These are nginx variables, not shell variables. The outer `REMOTESCRIPT` heredoc is also quoted, so Terraform's `${...}` interpolation is the only substitution that occurs — only `${var.openclaw_vm_password}` (and similar) are substituted at plan time.

#### Step 5 — Enable and start nginx
```bash
run_sudo ln -sf /etc/nginx/sites-available/openclaw /etc/nginx/sites-enabled/openclaw
run_sudo rm -f /etc/nginx/sites-enabled/default
run_sudo nginx -t
run_sudo systemctl enable nginx
run_sudo systemctl restart nginx
```

#### Step 6 — Health check loop
```bash
for i in {1..12}; do
  if curl -sf https://10.0.0.60/ > /dev/null 2>&1; then
    echo "==> HTTPS is up."
    exit 0
  fi
  sleep 5
done
echo "ERROR: HTTPS health check timed out."
journalctl -u nginx --no-pager -n 20 || true
exit 1
```

> After `mkcert -install`, the local CA is trusted by the system store (`update-ca-certificates` runs automatically), so `curl https://10.0.0.60/` works without `--cacert`.

### 5.2 New outputs in `terraform/outputs.tf`

```hcl
output "openclaw_https_url" {
  value       = "https://${var.openclaw_vm_host}/"
  description = "OpenClaw dashboard HTTPS URL"
}

output "rootca_mac_trust_commands" {
  value       = <<-EOT
    scp ${var.openclaw_vm_user}@${var.openclaw_vm_host}:/etc/ssl/openclaw/rootCA.pem ~/Downloads/openclaw-rootCA.pem
    sudo security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain ~/Downloads/openclaw-rootCA.pem
  EOT
  description = "Run these on your Mac after apply to trust the root CA in browsers"
}
```

---

## 6. Post-Apply — Trust Root CA on Mac

After `terraform apply` completes, run the two commands emitted by `rootca_mac_trust_commands`, then restart any open browsers. The padlock at `https://10.0.0.60/` should show as trusted.

---

## 7. Verification

```bash
# From Mac — should return 200
curl -sf -o /dev/null -w "%{http_code}" https://10.0.0.60/

# Confirm redirect
curl -sv http://10.0.0.60/ 2>&1 | grep "Location:"
# Expected: Location: https://10.0.0.60/
```

Open `https://10.0.0.60/` in Chrome or Safari — the dashboard should load with a valid padlock and the device-identity error should be gone.

---

## 8. Rollback

```bash
# On VM (or via a Terraform destroy of null_resource.openclaw_vm_nginx)
sudo systemctl stop nginx
sudo systemctl disable nginx
sudo apt-get remove -y nginx

# On Mac — remove trusted cert
sudo security delete-certificate -c "mkcert openclaw@10.0.0.60" /Library/Keychains/System.keychain
```

- [ ] nginx starts automatically on VM reboot (`systemctl is-enabled nginx` → `enabled`)
- [ ] `terraform apply` is idempotent — re-running does not break a working setup
